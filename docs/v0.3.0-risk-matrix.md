# sepigs v0.3.0 Risk Matrix

Status: active. M8.5 integrates experimental DNS publication alongside
Metrics/Dashboard/Router/Policy. Risks remain open until the full v0.3.0
acceptance matrix is complete.

Ratings use likelihood and impact values of Low, Medium, High, or Critical.
An item with Critical impact or High/High rating blocks promotion until its
gate is satisfied.

## Highest-Risk Decisions

1. Transaction commit semantics must be designed before modifying
   `Engine.reloadConfig`.
2. External binaries must be treated as untrusted test dependencies.
3. Plugin reload must not join the transaction until it has bounded and
   idempotent cleanup.
4. TUN, QUIC/Hysteria2, and WireGuard work must remain outside main.
5. A performance regression cannot be accepted solely to meet a feature
   milestone.

## Risk Review Cadence

- Review this matrix at each milestone exit.
- Add a risk when a new external process, privilege, persistent state, or
  public endpoint is introduced.
- Close a risk only with a linked test/report, not with implementation status.
- Reopen relevant v0.2 risks when a beta.1 backport is proposed.

## M6 Reload Design Review

- At M6, Metrics and Dashboard were the only experimental runtime participants.
  M7 later admitted Router/Policy under the explicit default-off allow-list.
- Router/policy is the proposed M7 scope because it can affect new connection
  decisions without migrating established streams. Atomic router/policy
  publication and health carry-over remain mandatory gates.
- DNS/fake-IP is split: resolver/cache work entered M8.5, while fake-IP runtime
  mapping migration remains M14 due to reverse-lookup continuity.
- Outbound active references and inbound drain/rebind are M9 design only.
- UDP generation ownership remains M14. Plugin/RPC/WASM reload remains a
  separate M11 threat and lifecycle review.
- R12/R16/R17 remain universal gates: each milestone needs bounded resources,
  environment-aware evidence, and no inflation of shadow/skipped results.

M6 reduces planning ambiguity but closes no runtime risk. A document is not
evidence that rollback or resource cleanup works.

## M7 Router/Policy Evidence

- R5/R6 are partially reduced for Router/Policy only: one immutable pair
  publishes atomically, rollback retains/restores the old pair, and a real
  established CONNECT stream survives while a new stream follows the
  candidate route.
- M7 does not reduce DNS/fake-IP, inbound/outbound, UDP, connection, or plugin
  runtime risks.
- The allow-list is `metrics/dashboard/router/policy`; legacy remains default.

## M8 DNS Design Review

- R8 remains open. The audit confirms ordinary cache and single-flight are
  resolver-instance state, while fake-IP mappings have separate pool/store
  ownership.
- DNS generation design binds cache, single-flight, DoH/UDP resources, and
  query completion to one immutable generation descriptor.
- Positive cache carry-over is conditional and never extends TTL. Upstream,
  route, strategy, or mode changes default to discard; negative cache is more
  restrictive.
- Commit does not cancel old queries. Late completion writes only to the owner
  generation cache.
- DoH/UDP public probing is disabled by default during prepare; local/mock
  health checks are bounded and redact endpoint material.
- Every fake-IP config difference rejects a DNS-only transaction. Mapping
  migration remains M10.
- DNS-specific risks and implementation tests are tracked in
  `dns-reload-risk-matrix.md`.

## M8.5 DNS Runtime Evidence

- R5 is partially reduced for DNS: prepare is side-effect free, commit switches
  one generation pointer, and rollback restores the old pointer and cache.
- R8 is partially reduced by generation-local cache/single-flight ownership,
  local old/new upstream smoke, and hard rejection of every fake-IP change.
- R12 remains open until the mixed DNS reload soak establishes cache and
  draining-generation memory bounds.
- R13 is guarded by unchanged `legacy` defaults and legacy reload regression
  tests.

At M8.5 the allow-list was `metrics/dashboard/router/policy/dns`. M11 adds
`outbound`, internally restricted to direct, block, and TCP relay. Fake-IP,
inbound, UDP sessions, connection-manager, protocol-specific/experimental
outbounds, and plugins remain excluded.

## M11 Limited Outbound Runtime Evidence

- R5/R6 are partially reduced for low-risk TCP outbounds: one registry pointer
  publishes, combined routing publication waits for outbound commit, and
  rollback restores the old generation.
- R7 is reduced for established TCP streams by generation references and
  drain-on-close behavior. It remains open for UDP and protocol-specific
  transports.
- R12 remains open until repeated-reload soak establishes generation/object
  memory bounds under long-lived streams.
- R13 is guarded by unchanged `legacy` default and regression tests.
- Secret-bearing medium/high-risk candidates reject without serialized config
  or credential values entering metrics/reports.
