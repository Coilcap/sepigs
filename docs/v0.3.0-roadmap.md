# sepigs v0.3.0 Roadmap

Status: planning only. This document does not authorize implementation,
version changes, tag creation, or release publication.

## Baseline

v0.3.0 starts from the published `v0.2.0-beta.0` evidence:

- The 24-hour soak passed with 14,645,605 successes, 11 errors
  (0.00007511%), p95 7.82 ms, 17,303 reloads, and final resources 0/0/0.
- P0 and P1 clients passed user manual signoff.
- CI, documentation, security, and beta packaging gates passed.
- External Shadowsocks/Trojan interoperability remains 0 verified,
  11 blocked, and 0 failed.
- Cross-component reload is not one transaction.

The first obligation of v0.3.0 is to preserve this baseline.

## Goals

1. Replace blocked external protocol claims with reproducible reference
   implementation evidence for protocols already present in sepigs.
2. Make configuration reload transactional across DNS, router, outbounds,
   inbounds, Dashboard, metrics, and plugins.
3. Establish a repeatable beta bugfix and backport process.
4. Strengthen UDP, fake-IP, DNS, Dashboard, and performance regression gates.
5. Promote only measured, non-breaking components from experimental to
   beta-ready.

## Non-Goals

v0.3.0 main must not include:

- A complete Hysteria2 implementation.
- A complete WireGuard implementation.
- Production TUN support.
- A large Dashboard redesign.
- A breaking configuration change or `configVersion` bump.
- A new protocol added only to increase protocol count.
- Claims of production-stable or v1.0 readiness.
- Movement or replacement of the published `v0.2.0-beta.0` tag.

## Milestones

### M0: Stability Fence

- Record v0.2 performance, soak, compatibility, and package baselines.
- Prepare the `release/v0.2.x` maintenance branch before the first v0.3
  runtime change lands.
- Define issue labels, severity, regression evidence, and backport rules.
- Freeze config version 1 and all current safe defaults.
- Produce a reload transaction design review before implementation.

Exit: maintenance and regression policy is reviewable, and no v0.3 change can
silently alter v0.2 behavior.

### M1: External Compatibility Evidence

- Pin supported reference implementation versions.
- Implement isolated launch/stop/timeout/log collection for
  shadowsocks-rust, shadowsocks-libev, sing-box, xray, and trojan-go where
  available.
- Run both sepigs-client/reference-server and
  reference-client/sepigs-server directions for implemented paths.
- Verify cipher, password failure, TLS/SNI, remote close, and large-payload
  cases.

Exit: required cases are verified with retained commands and logs, or a case
is explicitly unsupported. A local mock cannot satisfy this milestone.

### M2: Transactional Reload

- Introduce prepare, validate, commit, and abort phases.
- Stage DNS, router, outbound, inbound, metrics, Dashboard, and plugin changes.
- Keep the old runtime authoritative until every staged component is ready.
- Drain old inbounds only after commit.
- Roll back every staged component on injected failure.

Exit: fault injection at every stage leaves either the complete old state or
the complete new state, with no mixed state and no resource leak.

### M3: Stress And Surface Stabilization

- Run UDP/fake-IP/DNS traffic continuously through reload failure and success.
- Freeze and contract-test the Dashboard API used by the minimal Web UI.
- Establish v0.3 throughput, latency, RSS, heap, and event-loop baselines.
- Address only measured regressions.

Exit: P1 criteria pass without a config break or large UI expansion.

### M4: v0.3 Beta Closure

- Re-run the complete automated, compatibility, security, benchmark, soak,
  client, documentation, and packaging matrix.
- Update reality check and technical debt documents.
- Freeze runtime changes before the release candidate.
- Keep experimental branch results out of release claims.

Exit: release gates in `v0.3.0-acceptance-criteria.md` pass.

## First Implementation Phase

When implementation is separately authorized, begin with M0 and the launcher
portion of M1:

1. Create the maintenance boundary before runtime changes.
2. Pin reference binaries and define evidence formats.
3. Make one Shadowsocks and one Trojan happy-path launcher reproducible.
4. Add negative and cleanup cases.
5. Write the reload transaction design review.

Do not begin with TUN, QUIC, WireGuard, a Dashboard redesign, or protocol
expansion.

## Testing Strategy

- Every pull request runs the existing lint, typecheck, test, build,
  documentation, and security gates.
- External reference tests run in isolated jobs with pinned binaries, retained
  diagnostics, and strict process cleanup.
- Reload work uses stage-by-stage fault injection plus continuous TCP, UDP,
  DNS, fake-IP, Dashboard, and plugin probes.
- P1 stress work runs one-hour focused tests before joining the resumable
  release soak.
- Performance comparisons use repeated same-host samples and the budgets in
  `v0.3.0-acceptance-criteria.md`.
- Release candidates repeat client signoff for changed surfaces and run package
  inspection against an allowlist.
- Environment-blocked and product-failed results remain separate in every
  report.

## Release Strategy

The release path is evidence-driven:

1. `v0.3.0-alpha.N` may be proposed after P0 compatibility and reload work is
   integrated but before P1 promotion.
2. `v0.3.0-beta.N` requires all P0 gates, claimed P1 gates, long-soak evidence,
   updated reality check, and backward-compatible configuration.
3. `v0.3.0-rc.N` requires a runtime freeze, complete release matrix, clean
   package inspection, and no unresolved Critical/High release blocker.
4. A stable `v0.3.0` is not implied by this roadmap and requires separate
   authorization and production-readiness review.

Every prerelease tag is immutable, marked prerelease and not latest unless a
separate release decision says otherwise. The v0.2 tag and maintenance line
remain independent.

## Start Recommendation

Planning is approved. Implementation should start only after a separate user
instruction and only when M0 ownership, branch protection, and evidence
storage are agreed. The recommended v0.3.0 scope is feasible if P0 remains the
release gate and P2 remains isolated.
