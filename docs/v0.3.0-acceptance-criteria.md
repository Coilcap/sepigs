# sepigs v0.3.0 Acceptance Criteria

Status: M2 evidence complete; M4 isolated transactional prototype active.
These are promotion gates, not claims that transactional runtime integration
exists.

## Global Invariants

Every v0.3.0 candidate must satisfy:

- Node.js remains supported from `>=20.11.0`.
- ESM, strict TypeScript, lint, typecheck, tests, and build pass.
- Configuration remains version 1 and backward compatible.
- Safe defaults remain loopback-only and experimental features remain off.
- Existing v0.2 HTTP, SOCKS5, TCP, UDP, auth, routing, DNS, plugin, Dashboard,
  and lifecycle tests pass unchanged.
- No release artifact contains credentials, local absolute paths, soak
  checkpoints, profiles, or unapproved binaries.
- No P2 prototype is described as beta-ready or production-ready.

## P0: External Protocol Interoperability

Acceptance requires:

- Reference versions and artifact digests are recorded.
- Each launcher binds loopback only, uses a random port, has a hard timeout,
  captures stdout/stderr, and always terminates its process tree.
- Shadowsocks covers aes-128-gcm, aes-256-gcm, and
  chacha20-ietf-poly1305 with shadowsocks-rust or shadowsocks-libev plus at
  least one independent implementation such as sing-box.
- Trojan covers TLS, SNI/serverName, password rejection, remote close, and
  large payload with at least two available implementations among sing-box,
  xray, and trojan-go.
- Implemented inbound and outbound directions are tested separately.
- Wrong password fails without crashing or leaking plaintext.
- Large payload integrity is checked, not only connection establishment.
- Remote close releases all sockets, timers, listeners, and child processes.
- `npm run compat:external` produces machine-readable and Markdown reports.
- A result is `verified` only when the external process identity, command,
  version, assertions, and exit status are retained.
- Required release jobs have zero unexplained blocked/skipped cases.

## P0: Transactional Reload

Acceptance requires:

- Each reload has a revision/transaction ID and explicit prepare, validate,
  commit, drain, and abort states.
- DNS, router, outbound, inbound, metrics, Dashboard, and plugins expose
  bounded staging behavior.
- The active revision changes only after all required components prepare.
- A failure in any component leaves the complete old revision active.
- Abort is idempotent and releases staged sockets, servers, timers, plugin
  registrations, workers, and child processes.
- Old inbounds stop accepting only after commit; existing managed connections
  drain until completion or configured timeout.
- Same-address replacement limitations remain explicit and tested.
- Removed or changed plugins unload and unregister owner-scoped factories.
- Concurrent reload requests are serialized or rejected deterministically.
- Repeated file events coalesce without skipping the latest valid config.
- Fault injection covers every prepare, commit, drain, and abort boundary.
- At least 1,000 mixed successful/failed reloads complete with zero leaked
  resources and no mixed revision visible through Dashboard or metrics.
- Continuous HTTP, SOCKS5, UDP, DNS, fake-IP, and Dashboard probes meet the
  documented availability budget during reload.

### M3 Design Review Exit

- Current reload behavior and partial-commit risks are audited by component.
- The staged validate/prepare/health/commit/drain/rollback/cleanup lifecycle is
  documented.
- Generation and transaction state types compile under strict TypeScript.
- The component contract defines timeout and rollback-failure behavior.
- Dry-run loads and schema-validates current/candidate files, detects changed
  components, opens zero listeners, closes zero connections, and mutates no
  runtime state.
- Reload risk and observability contracts are reviewed.
- New state-machine, contract, timeout, and dry-run tests pass.
- Existing runtime tests and the M2 compatibility gate remain unchanged and
  passing.
- M3 completion permits an M4 prototype review; it does not permit direct
  replacement of `Engine.reloadConfig`.

### M4 Prototype Exit

- The executor follows the M3 state machine through parse, validate, prepare,
  health, commit, rollback, and cleanup.
- Prepare, health, commit, rollback, timeout, cancellation, event ordering,
  cleanup, and primary-error retention have deterministic fault tests.
- Ten prototype adapters implement the component contract without opening
  ports, closing connections, or mutating production runtime owners.
- Shadow mode loads validated config differences, executes the prototype
  transaction, and emits JSON and Markdown evidence with no runtime effects.
- Events are ordered, JSON serializable, bounded, and credential-redacted.
- Metrics are in-memory only and advance the active generation only on a
  successful shadow commit.
- Existing tests, security checks, and the M2 compatibility gate pass without
  lowering their baselines.
- M4 completion permits an M5 small-scope integration review. It does not
  permit a complete replacement of `Engine.reloadConfig`.

## P0: Beta Bugfix Pipeline

Acceptance requires:

- Issue templates capture version, platform, config redaction, reproduction,
  expected/actual behavior, logs, and severity.
- Regression labels distinguish runtime, security, compatibility,
  performance, packaging, and documentation defects.
- Every runtime fix adds or updates a regression test.
- Backports contain no feature work, config break, dependency expansion, or
  unrelated refactor.
- beta.1 triggers and exclusions match `beta-maintenance-policy.md`.
- Release notes can be generated from linked issues and verified commits.

## P1: UDP, Fake-IP, And DNS Reload Stress

Acceptance requires:

- Continuous mixed UDP/DNS/fake-IP traffic runs through successful and failed
  reloads.
- Fake-IP reverse mappings never resolve to the wrong real destination.
- Cache, negative cache, TTL, and single-flight bounds remain enforced.
- DNS failure fallback is deterministic and observable.
- High-cardinality and malformed DNS inputs remain bounded.
- One-hour stress passes with error rate below 0.1% and final resources 0/0/0.
- The v0.3 beta candidate includes these scenarios in the resumable long soak.

## P1: Dashboard API

Acceptance requires:

- The current endpoint set has contract fixtures and stable error shapes.
- Loopback-only default, bearer auth, minimum token length, rate limit,
  redaction, and CORS-off default remain enforced.
- Tokens, proxy credentials, target metadata outside the documented policy,
  and local paths do not enter logs or responses.
- Connection termination and reload requests are auditable and bounded.
- A failed reload leaves the Dashboard and previous runtime revision usable.
- API changes are additive; removals require a later config/API version plan.

## P1: Web Dashboard Beta-Ready Promotion

Acceptance requires:

- No large redesign and no new backend capability.
- UI uses only the stabilized Dashboard API contract.
- Read-only status remains usable when mutation permissions are unavailable.
- Disconnect/reload actions require explicit user confirmation and display the
  resulting revision/error.
- Token remains memory-only and is absent from URLs, storage, logs, and built
  assets.
- Current Chrome, Safari, and Firefox smoke tests cover loading, auth failure,
  reconnect, empty state, large connection lists, and API failure.
- Accessibility checks cover keyboard navigation, labels, focus, contrast,
  and reduced motion.
- The UI and API remain local-only by default.

## P1: Performance Baseline

Acceptance requires:

- Baselines record CPU, RSS, heap, event-loop delay, throughput, p50/p95/p99,
  errors, and final resource counters.
- Comparisons use the same host/runtime/config and at least five samples;
  report the median and variance.
- TCP/UDP throughput median does not regress more than 5% without approved
  evidence.
- p95 latency does not regress more than 10%.
- Steady-state RSS and heap p95 do not regress more than 10%.
- Final sockets/timers/listeners remain 0/0/0.
- A budget failure requires profiling and an explicit accept/reject decision.
- `npm run benchmark:gate`, `npm run soak:gate`, and the release soak pass.

## P2 Experimental Exit Criteria

P2 work stays off main. A prototype may request promotion only when:

- Its threat model, privilege model, cleanup/rollback behavior, dependency
  provenance, and platform support are documented.
- It has real packet/transport evidence rather than a mock-only result.
- It is disabled by default and absent from normal release behavior.
- It does not require a breaking config change.
- It passes a separate security and packaging review.

Meeting these criteria permits a promotion review; it does not guarantee a
main merge.

Module-specific research completion requires:

- Native TUN prototype: one explicitly supported OS can open/close a device in
  an isolated test, restore routes and DNS after failure, and document required
  privileges. It must not claim a production TCP/IP stack.
- QUIC/Hysteria2 research: dependency, ABI, packaging, security, congestion,
  and maintenance options are compared with a reproducible transport demo.
  A complete Hysteria2 implementation is outside scope.
- WireGuard adapter: external backend detection, key redaction, process
  timeout, interface cleanup, and route rollback are demonstrated. Active
  packet forwarding remains outside main.
- Rule-set ecosystem: format versioning, source trust, conflict behavior,
  bounded parsing, and large-set lookup costs are measured before any public
  ecosystem claim.
- Subscription provider updater: SSRF controls, size/time limits, signature or
  trust policy, atomic cache replacement, rollback, and disabled-by-default
  behavior are designed before remote fetching is considered.

## Release Gate

Before a v0.3.0 beta prerelease:

- P0 criteria pass.
- Selected P1 criteria claimed in release notes pass.
- Full lint, typecheck, test, build, docs, security, compatibility,
  benchmark, soak, client, and package checks pass.
- A fresh reality check identifies beta-ready, experimental, and unsupported
  modules.
- Release notes state that the release is beta and not production-stable.
- The v0.2 maintenance line and published tag remain unchanged.
